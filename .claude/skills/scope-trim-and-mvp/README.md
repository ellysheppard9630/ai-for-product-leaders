# Scope Trim and MVP

## Purpose
Reduces a feature, project, or product idea to a more realistic and valuable MVP. Analyzes the proposal from five perspectives — customer value, customer friction, product prioritization, engineering practicality, and architecture complexity — then synthesizes a trimmed scope recommendation with explicit phasing and tradeoffs.

## Use Cases
- A feature proposal is directionally good but feels too large, too vague, or too expensive to deliver confidently in its current form
- A roadmap item has too many stakeholder asks layered on top of each other and needs a clear first phase defined
- You want to define what belongs in MVP versus what belongs in a later phase, with reasoning

## Inputs
Any of the following:
- A rough feature idea or concept
- A PRD or feature spec
- A project summary or roadmap description
- A stakeholder ask or customer request
- A pasted document fragment

## Outputs
A structured document covering:
1. Proposal summary
2. Core customer value (what must be preserved)
3. Customer friction and objections (what can be cut)
4. Product scope review
5. Engineering practicality review
6. Architecture complexity review
7. Recommended MVP (minimum viable workflow and must-haves)
8. What to defer, phase, or drop
9. Tradeoffs and residual risks
10. Recommended next step
